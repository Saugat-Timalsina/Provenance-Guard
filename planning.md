# Provenance Guard — Engineering Planning Document

---

## Project Overview

### Purpose

Provenance Guard is a backend API that analyzes submitted text and attempts to determine whether it was written by a human or generated by an AI model. It does not make a final authoritative judgment. Instead, it combines multiple detection signals, produces a calibrated confidence score, and presents the result in plain language that any user can understand.

The system is designed for a writing platform context — a place where readers, editors, and creators need to understand where content came from. The goal is not to punish AI-assisted writing, but to make the origin of content visible and contestable.

### Project Goals

1. Accept text submissions via a REST API and return a structured analysis result.
2. Use two independent detection signals to analyze each submission — one based on AI reasoning, one based on measurable text properties — so that neither signal alone determines the outcome.
3. Express the result as a confidence score and a plain-language transparency label that a non-technical reader can act on.
4. Allow creators to file appeals if they believe a label is wrong, and track those appeals in a structured audit log.
5. Protect the API from abuse with rate limiting tuned to realistic usage patterns.
6. Log every significant action in a structured format so the system's behavior is auditable.

## Architecture Narrative

Every text submission travels through a fixed pipeline. Each stage either enriches the result or rejects the request. Nothing is skipped.

When a creator disagrees with a label, they file an appeal. The appeal does not re-run the detection pipeline — it flags the submission for human review and updates its status.

## Detection Signals

### Signal 1 — Groq LLM Classification

#### What It Measures

This signal asks a large language model (via the Groq API) to read the submitted text and assess the likelihood that it was generated by an AI. The LLM is prompted to look for patterns that AI writing tools commonly produce: overly uniform sentence rhythm, the absence of personal anecdote or error, unnaturally smooth transitions, and the tendency to cover all sides of a topic without committing to a position.

The output is a probability score between 0.0 and 1.0, where 1.0 means "almost certainly AI-generated" and 0.0 means "almost certainly human-written." The LLM also returns a brief natural-language explanation for its judgment, which is passed through to the API response.

#### Why It Was Selected

A large language model is the only detection tool that can reason about the *meaning and intent* of text, not just its surface statistics. It can recognize when a passage reads like a generic synthesis rather than a personal observation. It can notice when technical explanations are unusually complete for a first draft. These are things that pure statistical analysis cannot capture.

#### Strengths

- Captures semantic coherence and rhetorical patterns that are invisible to heuristics.
- Can reason about context — a formal essay and a casual message are evaluated differently.
- Returns a human-readable explanation that can be shown directly to users.
- Adapts implicitly to newer AI writing styles as the underlying model is updated.

#### Weaknesses

- **Not deterministic.** The same text may receive slightly different scores across calls due to model temperature and sampling. This is why it is combined with a stable heuristic signal.
- **Biased against formal writing.** Academic writing, legal documents, and professional reports often score high for AI likelihood even when written by humans, because they share structural properties with AI output: complete sentences, organized paragraphs, hedged language.
- **Cannot detect lightly edited AI text.** If a human revises AI-generated output heavily enough, this signal may classify it as human.
- **Dependent on external API availability.** The Groq API is a network dependency. If it is unavailable, this signal cannot produce a score.

#### Output Format

```
{
  "signal": "llm_classification",
  "score": 0.82,
  "explanation": "The text exhibits uniform sentence structure and covers
                  multiple perspectives without personal voice, which is
                  characteristic of AI-generated writing."
}
```

### Signal 2 — Stylometric Heuristics

Stylometry is the study of measurable style properties in text. Unlike the LLM signal, this signal does not require an API call — it analyzes the text locally using three quantitative metrics. Each metric captures a different dimension of writing style.

#### Metric 1: Sentence Length Variance

**What it captures:** Human writers vary their sentence length naturally — a complex thought gets a long sentence, a punch line gets a short one. AI-generated text tends to produce sentences of similar length throughout a passage, because the model optimizes for coherent output rather than rhetorical rhythm. Low variance in sentence length is a weak but real signal of machine authorship.

**Why it helps:** It is computationally cheap, deterministic, and independent of the LLM signal. It measures a structural property of the text that is largely invisible to a casual reader but statistically consistent across AI-generated samples.

**Weaknesses:** Technical writing (instruction manuals, legal clauses) and news writing naturally have low sentence length variance regardless of authorship. This metric alone would misclassify those genres. It is a supporting signal, not a primary one.

#### Metric 2: Type-Token Ratio (TTR)

**What it captures:** Type-Token Ratio is the number of unique words (types) divided by the total number of words (tokens). A high TTR means the writer uses a wide variety of vocabulary. A low TTR means words are repeated frequently. AI-generated text often scores in a middle band — it avoids extreme repetition but also avoids rare or idiosyncratic vocabulary choices.

**Why it helps:** Human writers at the extremes — highly specialized experts or casual conversationalists — produce TTR values that AI models rarely replicate. An expert uses domain-specific jargon that appears rarely in training data. A casual writer repeats filler words naturally. Both patterns are detectable.

**Weaknesses:** TTR shrinks as text gets longer, even for the same author. A 5,000-word essay will always have a lower TTR than a 200-word paragraph. This metric is most reliable for texts of similar length. Very short submissions (under 50 words) will produce TTR values too unstable to interpret.

#### Metric 3: Punctuation Density

**What it captures:** The ratio of punctuation characters to total characters in the text. AI models tend to use punctuation conventionally and conservatively — they almost never use ellipses for dramatic effect, rarely use em-dashes as parenthetical interruptions, and avoid exclamation marks in analytical writing. Human writers, especially in informal or expressive contexts, punctuate more erratically.

**Why it helps:** This metric catches stylistic signatures that the LLM signal might normalize. A human blogger who writes "wait — what?" reads very differently from AI output. Punctuation density, combined with the other two metrics, adds texture to the stylometric profile.

**Weaknesses:** Formal human writing (academic papers, legal memos) uses punctuation as conservatively as AI output. This metric is most useful for informal or mixed-register text, and least useful for structured professional documents.

#### Combined Stylometric Score

The three metrics are each normalized to a 0.0–1.0 range based on empirically observed ranges for AI-generated text versus human text. They are then averaged into a single stylometric score:

```
stylometric_score = (
    sentence_length_variance_score * 0.40 +
    type_token_ratio_score         * 0.35 +
    punctuation_density_score      * 0.25
)
```

Sentence length variance receives the highest weight because it is the most consistent predictor across text types. Punctuation density receives the lowest weight because it is the most genre-dependent.

#### Output Format

```
{
  "signal": "stylometric_heuristics",
  "score": 0.61,
  "components": {
    "sentence_length_variance": { "raw": 4.2, "score": 0.55 },
    "type_token_ratio":         { "raw": 0.71, "score": 0.63 },
    "punctuation_density":      { "raw": 0.034, "score": 0.67 }
  }
}
```

## Confidence Scoring

### Design Philosophy

The confidence score is not a vote between two signals. It is a weighted combination that reflects how much we trust each signal given its known strengths and weaknesses.

The LLM classification signal receives a higher weight (0.65) because it reasons about meaning, not just surface statistics. The stylometric signal receives a lower weight (0.35) because it is genre-sensitive and can be fooled by writing style alone.

```
confidence_score = (llm_score * 0.65) + (stylometric_score * 0.35)
```

This formula produces a single score between 0.0 and 1.0, where values near 1.0 indicate high confidence that the text is AI-generated, and values near 0.0 indicate high confidence that it is human-written. Values near 0.5 indicate genuine uncertainty — the signals disagree, or neither is confident.

### Why False Positives Are Worse Than False Negatives

In this system, a false positive means labeling a human-written text as AI-generated. That is a more serious error than a false negative (missing AI text), because:

1. It directly harms a creator's reputation and may result in their content being flagged or removed.
2. It is harder to appeal effectively — a human who genuinely wrote something may have difficulty proving it.
3. It erodes trust in the platform faster than occasional AI content slipping through.

The threshold design reflects this asymmetry. The boundary for the "Likely AI" label is set at 0.70 — a high bar — so that only text that both signals agree is AI-generated with substantial confidence receives the strongest label.

### Uncertainty Representation

Uncertainty is not a failure state. It is an honest acknowledgment that the two signals are either in conflict or that neither has reached a confident conclusion. The system treats uncertainty as its own outcome, not as a default when detection fails.

A score in the 0.40–0.69 range is not "probably AI but we're not sure." It is a meaningful result that says: "This text has properties that could belong to either origin, and we cannot responsibly label it either way."

Confidence should never be communicated to users as certainty. Even a score of 0.95 does not mean the text is definitely AI-generated — it means the signals agree strongly. This distinction is reflected in the label language.

### Thresholds

| Score Range | Label Category | Reasoning |
|-------------|---------------|-----------|
| 0.00 – 0.39 | Likely Human  | Both signals lean toward human authorship. The bar is set at 0.39 rather than 0.50 to avoid the center of the uncertain zone. |
| 0.40 – 0.69 | Uncertain     | The signals disagree, or neither reaches a confident conclusion. A 30-point band gives the uncertain region room to capture genuine ambiguity. |
| 0.70 – 1.00 | Likely AI     | Both signals lean strongly toward AI authorship. The 0.70 floor reflects the asymmetry principle — we require high agreement before applying the most consequential label. |

### Threshold Validation Approach

Before finalizing thresholds, the following test inputs will be used to confirm that scores behave as expected:

- A Wikipedia article introduction (formal, encyclopedic, expected: Uncertain to Likely AI)
- A personal blog post with informal voice and first-person anecdotes (expected: Likely Human)
- Raw ChatGPT output with no editing (expected: Likely AI)
- A short academic abstract (expected: Uncertain — formal human writing)
- A news article (expected: Uncertain — structured but human-written)
- A casual text message (expected: Likely Human — irregular punctuation, short sentences, repetition)

If any of these inputs produces a label that is clearly wrong, the weights or thresholds will be adjusted before the system is considered production-ready.

## Transparency Labels

The following label texts are the exact strings shown to users in the API response and audit log. They are written for a non-technical reader. No jargon is used. Each label explains both the result and its limitations.

### High-Confidence AI Label
*(score: 0.70 – 1.00)*

> "Our analysis strongly suggests this text was generated by an AI writing tool. Both the writing style and the structural patterns are consistent with AI-generated content. This is not a definitive ruling — if you believe this is incorrect, you can file an appeal."

### Uncertain Label
*(score: 0.40 – 0.69)*

> "We were unable to determine with confidence whether this text was written by a human or generated by an AI tool. The writing shows a mix of patterns from both. No label has been applied. If you believe your content deserves a clearer classification, you can file an appeal."

### High-Confidence Human Label
*(score: 0.00 – 0.39)*

> "Our analysis suggests this text was most likely written by a human. The writing style and structure are consistent with human authorship. This result is based on automated signals and may not be fully accurate for all writing styles."

### Design Notes

- All three labels use "suggests" or "unable to determine" rather than "is" or "proves." This language choice is intentional — it communicates probability, not fact.
- All three labels mention the possibility of appeal or limitation, so users in all categories understand the system is not a final arbiter.
- The labels differ by wording, not just by the score number embedded in them. A grader reading the label alone can identify which confidence tier it belongs to without seeing the score.

## Appeals Workflow

### Who Can Appeal

Any submitter who receives a label they disagree with may file an appeal. There is no authentication requirement in the current implementation — appeals are identified by `submission_id`. In a production system, this would be tied to a creator account.

### Required Fields

An appeal submission must include:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `submission_id` | integer | Yes | The ID of the submission being appealed |
| `reason` | string | Yes | The creator's explanation for why the label is incorrect. Minimum 10 characters. |

### Status Lifecycle

```
[filed] → "pending"
       → "under_review"   (set immediately when appeal is created)
       → "resolved"       (set by a reviewer — outside current scope)
       → "rejected"       (set by a reviewer — outside current scope)
```

When an appeal is created, the submission's status field is updated to `"under_review"` immediately. This ensures that any platform logic checking submission status knows not to treat the original label as final.

### What Gets Logged

Every appeal creates two audit log entries:

1. **Appeal Filed** — includes the `submission_id`, the original label, the original confidence score, the appeal reason, and the timestamp.
2. **Status Change** — records that the submission moved from its original status to `"under_review"`.

The audit log is designed so that a reviewer can read both entries side by side and understand the full history of a submission without querying any other table.

### What a Reviewer Would See

A reviewer querying `GET /log` for a specific submission would see:

1. The original `submission_analyzed` event with attribution result, confidence score, signal breakdown, and timestamp.
2. The `appeal_filed` event with the creator's reasoning and the timestamp of the appeal.
3. Optionally, a `status_changed` event confirming the transition to `under_review`.

This sequence gives a reviewer everything they need to make a decision: what the system said, when it said it, and what the creator disputes.

### Anticipated Edge Cases

**Edge Case 1: Appeal on an already-appealed submission**
A creator submits two appeals for the same submission. The system should reject the second appeal with a `409 Conflict` response and a message explaining that an appeal is already pending. The first appeal must not be overwritten.

**Edge Case 2: Appeal on a submission that does not exist**
If `submission_id` does not match any record in the database, the system returns `404 Not Found`. This prevents appeals from being filed against phantom IDs.

## API Design

### Endpoint 1: POST /submit

**Purpose:** Accept a text submission, run the detection pipeline, and return a structured result.

**Request:**
```
POST /submit
Content-Type: application/json

{
  "text": "The mitochondria is the powerhouse of the cell. It produces ATP
           through a process called cellular respiration..."
}
```

**Successful Response (200 OK):**
```
{
  "submission_id": 42,
  "submitted_at": "2026-06-29T14:23:01Z",
  "label": "Likely AI",
  "confidence": 0.81,
  "explanation": "Our analysis strongly suggests this text was generated
                  by an AI writing tool...",
  "signals": {
    "llm_classification": {
      "score": 0.87,
      "explanation": "Uniform sentence structure and comprehensive coverage
                      without personal voice."
    },
    "stylometric_heuristics": {
      "score": 0.69,
      "components": {
        "sentence_length_variance": { "raw": 3.1, "score": 0.72 },
        "type_token_ratio":         { "raw": 0.68, "score": 0.65 },
        "punctuation_density":      { "raw": 0.028, "score": 0.71 }
      }
    }
  }
}
```

**Error Responses:**
- `400 Bad Request` — text field is missing or empty
- `400 Bad Request` — text exceeds maximum length (10,000 characters)
- `429 Too Many Requests` — rate limit exceeded
- `503 Service Unavailable` — Groq API is unreachable

### Endpoint 2: POST /appeal

**Purpose:** File an appeal against an existing submission's label.

**Request:**
```
POST /appeal
Content-Type: application/json

{
  "submission_id": 42,
  "reason": "I wrote this myself. It reflects my own research and uses
             my natural writing style. I believe the system is penalizing
             my use of clear, organized writing."
}
```

**Successful Response (201 Created):**
```
{
  "appeal_id": 7,
  "submission_id": 42,
  "status": "pending",
  "filed_at": "2026-06-29T14:45:11Z",
  "message": "Your appeal has been received. The submission is now marked
              as under review. A human reviewer will examine your case."

```

**Error Responses:**
- `400 Bad Request` — missing `submission_id` or `reason`
- `400 Bad Request` — `reason` is fewer than 10 characters
- `404 Not Found` — `submission_id` does not exist
- `409 Conflict` — an appeal for this submission already exists
- `429 Too Many Requests` — rate limit exceeded

## Edge Cases

The following scenarios represent realistic situations where the system is likely to produce an incorrect or misleading result. These are documented in advance so that the Known Limitations section of the README reflects genuine signal-level analysis rather than generic disclaimers.

### Edge Case 1: Academic and Professional Writing

**Scenario:** A PhD student submits a section of their thesis. The writing is formal, well-organized, uses passive voice, avoids first-person, and covers multiple perspectives without expressing a strong opinion.

**Why the system struggles:** Both signals will flag properties consistent with AI generation — the LLM will note uniform sentence structure and comprehensive coverage; the stylometric signal will find low variance and moderate TTR. The result may be "Likely AI" or "Uncertain" for genuinely human-written academic prose.

**Signal explanation:** This is a known weakness of every AI detection system. Academic writing conventions overlap heavily with AI writing conventions because academic writing prioritizes clarity and completeness over voice and spontaneity.

### Edge Case 2: Very Short Submissions

**Scenario:** A user submits a 20-word sentence or a single paragraph.

**Why the system struggles:** The stylometric signal requires enough tokens to produce stable variance measurements. With fewer than 50 words, sentence length variance is based on too few data points to be meaningful. TTR is artificially inflated (short texts always have high type diversity). The system will likely return an "Uncertain" result with a note about insufficient length.

**Planned behavior:** For texts under 50 words, the stylometric signal weight will be reduced and the response will include a warning that the result is less reliable for very short texts.

### Edge Case 3: Heavily Edited AI Text

**Scenario:** A creator generates a draft with ChatGPT and then rewrites it substantially — changing sentence structures, adding personal examples, removing formal transitions.

**Why the system struggles:** After sufficient human editing, the stylometric fingerprint of the AI output is partially overwritten. The LLM signal may still detect AI-like patterns in the underlying structure, but the surface-level statistics will shift toward human. The result will likely be "Uncertain."

**Why this is acceptable:** A heavily edited AI-assisted text occupies a genuine middle ground. It is neither purely human nor purely AI. An "Uncertain" label is arguably the most accurate result in this case.

### Edge Case 4: Code and Structured Non-Prose Text

**Scenario:** A user submits a block of code, a JSON document, a numbered list, or a table in markdown.

**Why the system struggles:** The stylometric metrics are designed for natural language prose. Punctuation density is meaningless for code (which is full of brackets, semicolons, and symbols). Sentence length variance cannot be computed meaningfully for structured data. The LLM may also struggle to classify code as human or AI because the stylistic conventions of code are different from prose.

**Planned behavior:** The system will process these submissions but the accuracy is expected to be low. The README will name this as a specific known limitation.

### Edge Case 5: Non-English Text

**Scenario:** A user submits text in French, Spanish, Arabic, or another language.

**Why the system struggles:** The Groq LLM prompt is written in English and instructs the model to evaluate English text properties. The stylometric metrics were calibrated on English text corpora. For other languages, sentence boundaries, punctuation conventions, and vocabulary distribution are different. Results for non-English text are unreliable.

**Planned behavior:** The system will not reject non-English text, but the README will document that results are only reliable for English-language submissions.

### Edge Case 6: Intentionally Adversarial Text

**Scenario:** A user deliberately crafts text to evade detection — introducing spelling errors, using unusual punctuation, or mimicking human conversational style.

**Why the system struggles:** The stylometric signal measures surface properties and can be fooled by deliberate manipulation. The LLM signal is harder to fool but not immune — if adversarial text mimics the kind of informal, idiosyncratic human writing that the LLM associates with human authorship, it may be misclassified.

**Why this is acceptable:** No AI detection system is adversarially robust. The system's goal is accurate classification of ordinary text, not detection of motivated evasion.
